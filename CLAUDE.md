# compliance-tracker — Claude Code Context

## Project
Full-stack GraphQL compliance task tracker. See PLAN.md for full feature scope and progress.

## Stack
- **Server**: Node.js + TypeScript + Apollo Server 4 + Express, port 4000
- **Client**: React + TypeScript + Apollo Client + MUI + Vite
- **Database**: MongoDB via Mongoose (Docker)
- **Testing**: Jest (server resolvers) + React Testing Library (simple components) + Playwright (planned, real-browser e2e)

## General principles
- Always follow the official migration guides and documented recommendations from the stack vendors (MUI, Apollo, Vite, etc.) — prefer their prescribed patterns over workarounds, even if the workaround is shorter

## Key conventions
- Schema lives in `server/src/schema/schema.graphql` (SDL file, loaded with `readFileSync`)
- Every Mongoose query in a resolver must end with `.exec()` to return a real Promise (Mongoose 7+ re-execution guard)
- A fresh `ownerLoader` DataLoader is created per-request in the Apollo context factory to ensure per-request batching
- `dotenv` is configured to load `.env.local` explicitly — not `.env`
- `setup` npm script handles copying `.env.local.dist` → `.env.local` (cross-platform Node inline, run once after clone)
- npm scripts must use cross-platform syntax — no `[ -f ... ]` bash conditions, use `node -e` inline instead
- Use `undefined` over `null` for optional values; use `?` optional marker on props/params rather than `string | undefined`
- GraphQL queries and mutations live in `.graphql` files under `src/graphql/` — never inline `gql` strings
- Client TypeScript types are generated by codegen from `schema.graphql` — never write them manually
- One component per file — including styled layout components; no barrel files bundling multiple components together
- Use `@/` absolute imports for all intra-project imports (e.g. `@/components/Foo`, `@/graphql/__generated__/types`) — configured in `tsconfig.json` paths, `vite.config.ts` resolve alias, and `jest.config.ts` moduleNameMapper
- Do not `import React from 'react'` — the automatic JSX transform (`jsx: react-jsx`) handles this; import only named exports that are actually used (e.g. `import { Fragment } from 'react'`)

## Environment
- Platform: Windows + Git Bash
- MongoDB runs in Docker: `docker compose up -d` from project root
- Server and client dev servers are managed via IntelliJ run configurations — do not attempt to start/stop/kill them from the terminal

## Running locally
```bash
# First time only
cd server && npm run setup
cd client && npm run setup

# Start MongoDB
docker compose up -d

# Start server
cd server && npm run dev
# → http://localhost:4000/graphql
# → http://localhost:4000/health

# Start client
cd client && npm run dev
# → http://localhost:5173

# Regenerate GraphQL types after schema changes
cd client && npm run codegen
```

## Testing and sign-off
- Run `npx tsc --noEmit` in both `client/` and `server/` and confirm zero errors before considering any change complete — in `client/` also run `npx tsc -p tsconfig.node.json --noEmit` to catch errors in `vite.config.ts` and `codegen.ts` (outside the main `src/` include)
- Run `npm test` in both packages and confirm all tests pass
- UI changes must be verified visually by the user in the running app before the corresponding PLAN.md checkbox is marked done — do not mark UI milestones complete based on passing unit tests alone
- Manual test steps for each UI feature live in `docs/features/<feature-name>.md` — create one whenever a new UI feature is added

## Commit style
One commit per major milestone:
1. `feat: scaffold server — Apollo Server 4, schema, resolvers, DataLoader, DateTime scalar`
2. `feat: scaffold client — Vite + React + Apollo Client + MUI + codegen`
3. `feat: UI — controls dashboard, task drawer, optimistic updates`
4. `feat: tests — resolver unit tests + RTL component tests`
