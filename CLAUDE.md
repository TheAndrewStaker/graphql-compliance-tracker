# compliance-tracker — Claude Code Context

## Project
Full-stack GraphQL compliance task tracker. See PLAN.md for full feature scope and progress.
See docs/ARCHITECTURE.md for system structure and data flow.

## Stack
- **Server**: Node.js + TypeScript + Apollo Server 5 + Express 5, port 4000
- **Client**: React 19 + TypeScript + Apollo Client 4 + MUI 7 + Vite 7
- **Database**: MongoDB via Mongoose 9 (Docker)
- **Testing**: Jest (server resolvers) + React Testing Library (components) + Playwright (planned)

## General principles
- Always follow the official migration guides and documented recommendations from the stack vendors (MUI, Apollo, Vite, etc.) — prefer their prescribed patterns over workarounds, even if the workaround is shorter
- Prefer the latest compatible version of every package — when adding a dependency or upgrading, use the current stable release unless a specific version constraint is documented here

## Key conventions
- Every Mongoose query in a resolver must end with `.exec()` to return a real Promise
- DataLoader instances are created per-request in the Apollo context factory — never share them across requests
- `dotenv` loads `.env.local` explicitly — not `.env`
- `setup` npm script handles copying `.env.local.dist` → `.env.local` (cross-platform Node inline, run once after clone)
- npm scripts must use cross-platform syntax — no `[ -f ... ]` bash conditions, use `node -e` inline instead
- Use `undefined` over `null` for optional values; use `?` optional marker on props/params rather than `string | undefined`
- GraphQL operations live in `.graphql` files under `client/src/graphql/`, one file per model (e.g. `controls.graphql`, `owners.graphql`, `tasks.graphql`) — never inline `gql` strings, never a single flat `queries.graphql` or `mutations.graphql`
- Use GraphQL fragments with fragment masking (codegen `client` preset default) for all models that have both a list view and a detail/edit view:
  - Define two fragments per model in the model's `.graphql` file: `<Model>ListFields` (columns shown in a DataGrid or list) and `<Model>EditFields` (all fields needed by the create/edit form, including nested ids for selects)
  - Spread both in every query and mutation that returns that model — e.g. `GetTasks { tasks { ...TaskListFields ...TaskEditFields } }`
  - Codegen generates `<Model>ListFieldsFragment` and `<Model>EditFieldsFragment` — import these directly instead of indexing into query types (`GetTasksQuery['tasks'][0]`)
  - Page components receive the opaque masked row from the query and pass it to dialogs; dialog components call `useFragment(ModelEditFieldsFragmentDoc, task)` at the top of the component to unmask
  - In page-level callbacks (DataGrid `valueGetter`, `getActions`) where ESLint's `react-hooks/rules-of-hooks` would falsely flag the `use`-prefixed codegen utility, alias it at the import: `import { useFragment as readFragment } from '@/graphql/__generated__/fragment-masking'`
  - For optimistic responses on masked mutation return types, cast through `unknown`: `{ ... } as unknown as CreateTaskMutation['createTask']`
  - Fragment masking utilities (`useFragment`, `FragmentType`, `makeFragmentData`) come from `@/graphql/__generated__/fragment-masking`; generated types and document nodes come from `@/graphql/__generated__/graphql`
- Server resolvers are split by model (e.g. `control.resolver.ts`) and merged in `resolvers/index.ts`
- Client TypeScript types are generated by codegen from `schema.graphql` — never write them manually; import from `@/graphql/__generated__/graphql`
- Components import `useQuery`/`useMutation` from `@apollo/client/react` and pass typed document nodes — no generated hook wrappers
- `ApolloProvider` comes from `@apollo/client/react`; `MockedProvider` (tests) from `@apollo/client/testing/react`
- One component per file — including styled layout components; no barrel files bundling multiple components together
- Use `@/` absolute imports for all intra-project imports — configured in `tsconfig.json` paths, `vite.config.ts` alias, and `jest.config.ts` moduleNameMapper
- Do not `import React from 'react'` — the automatic JSX transform handles this; import only named exports actually used

## Environment
- Platform: Windows + Git Bash
- MongoDB runs in Docker: `docker compose up -d` from project root
- Server and client dev servers are managed via IntelliJ run configurations — do not attempt to start/stop/kill them from the terminal

## Running locally
```bash
# First time only
cd server && npm run setup
cd client && npm run setup

# Start MongoDB
docker compose up -d

# Start server (via IntelliJ) → http://localhost:4000/graphql
# Start client (via IntelliJ) → http://localhost:5173

# Regenerate GraphQL types after schema changes
cd client && npm run codegen

# Validate all .feature files
npm run validate:features
```

## Testing and sign-off
- Run `npx tsc --noEmit` in both `client/` and `server/` and confirm zero errors before considering any change complete
- Run `npm test` in both packages and confirm all tests pass
- UI changes must be verified visually by the user in the running app before the corresponding PLAN.md checkbox is marked done — do not mark UI milestones complete based on passing unit tests alone
- Manual test steps for UI features live in `docs/testing/manual/<feature-name>.md`

## Session lifecycle
**Starting a session:** Run `/session:new` first to populate `SESSION_CURRENT.md` with today's focus and carry-over context before any implementation work begins.

**Closing a session**, in this order:
1. Complete all implementation work and confirm `npx tsc --noEmit` + `npm test` pass
2. Update all docs: `tasks.md` checkboxes, `PLAN.md` status and decisions log, `ARCHITECTURE.md` if patterns changed
3. Update `SESSION_CURRENT.md` to reflect what was actually completed this session — the close script archives the file as-is, so this must be done before closing
4. Run `/session:close` to archive `SESSION_CURRENT.md` → `docs/sessions/YYYY-MM-DD.md`
5. Commit — the archived session file is included in the commit so the session is a closed artifact in git history, not an in-progress one
